Actual provision.sh flow (verified against scripts/provision.sh):

1. Parse `./coolify.yaml`, validate with `bash $SCRIPT_DIR/validate.sh`, bail on error.
2. `coolify_load_server "$SERVER_ALIAS"` + `doppler_load_account "$SERVER_ALIAS"`.
3. `coolify_upsert_project "$PROJECT" "..."` → PROJECT_UUID.
4. Read `server_name` from `~/.claude/coolify.json` (default `localhost`), then `coolify_get_server_uuid "$SERVER_NAME"` → SERVER_UUID.
5. `coolify_get_destination_uuid "$SERVER_UUID"` → DEST_UUID (optional).
6. Read `ssh_host` from coolify.json (REQUIRED, no fallback — bail if missing).
7. For each ENV in [staging, production]:
   - `coolify_find_app_by_name "$APP_NAME"` → APP_UUID, or POST to create.
   - PATCH app settings (fqdn, auto_deploy=off, custom_docker_run_options=mount, health_check_path=/api/health).
   - SSH to server and `docker volume create ${APP_UUID}-doppler-cache`.
   - Revoke old Doppler service token, create new one via `doppler_create_service_token`.
   - Bulk-set env vars on the app (DOPPLER_TOKEN + every key in env_vars, all RUNTIME — values fetched from Doppler via subprocess; fails the whole run on any doppler-get failure).
   - Verify volume-mount round-trip — hard fail if `custom_docker_run_options` did not persist.
8. Write back `coolify_app_ids` to `coolify.yaml`.
9. END — provision.sh does NOT call coolify_deploy_app. First deploy is triggered by pushing to `main`, which runs the generated `.github/workflows/deploy.yml`.

Functions NOT called by provision.sh:
- `coolify_get_github_app_uuid` (defined in lib-coolify-api.sh but dead code)
- `coolify_deploy_app` (defined in lib-coolify-api.sh; only called by test/e2e.sh)
