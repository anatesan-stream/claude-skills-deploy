<objective>
Fix BUG-02: the Python heredoc in provision.sh that fetches per-key secrets from Doppler does not check `result.returncode`. When `doppler secrets get` fails (network blip, revoked token, wrong project), `result.stdout` is empty, the empty string is appended to ENVS_JSON, and Coolify silently receives an empty env var.

Purpose: Surface Doppler failures loudly. The operator must see WHICH key failed and WHY (Doppler stderr) so they can diagnose without re-running.

Output: Modified Python heredoc in provision.sh that loops through all env_vars, accumulates `(key, stderr)` pairs on failure, and raises SystemExit with a clear multi-line error message if any key failed.
</objective>

Current state (scripts/provision.sh lines 147-161):
The Python heredoc reads project, config, token, env_vars_str from sys.argv[1:5], then for each key runs `doppler secrets get ... --plain` via subprocess.run and unconditionally appends `result.stdout.strip()` to the data list.

The bug: `v = result.stdout.strip()` ignores `result.returncode` and `result.stderr`.

Established patterns in the codebase:
- Fail-fast error format used elsewhere: `ERROR: <field> ...` written to stderr
- `set -euo pipefail` is active — when Python raises SystemExit inside `$(python3 - ... <<PY ... PY)`, the command substitution exits non-zero and `set -e` aborts provision.sh
- Python errors must be written to stderr (not stdout) because stdout is captured into $ENVS_JSON
- The `<<'PY'` quoted delimiter prevents shell expansion inside the Python body — preserve this
</interfaces>
